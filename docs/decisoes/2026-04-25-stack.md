# Stack tecnologica — Next.js + FastAPI + Supabase + QStash

**Data:** 2026-04-25
**Status:** aceita
**Tags:** decisao, stack, arquitetura, mvp

## Contexto

Precisamos definir stack para SaaS multitenant que: recebe upload de audio, converte pra MP3, analisa via IA, gera 3 variacoes de copy, posta no YouTube. MVP em 30 dias, executor principal e Gustavo (em formacao). Decisao precisa balancear: velocidade de MVP, custo de infra, alinhamento com stack que o Henrique ja domina (pra suportar Gustavo), e capacidade de escalar pra Nivel 3 sem reescrita.

## Opcoes Consideradas

### 1. Reaproveitar Yabadoo Desktop (transcribe) — Python + pywebview + Svelte
- **Pros:** 80% do codigo ja existe (watchdog, pluggy, multi-provider AI, usage_tracker)
- **Contras:** Desktop Windows-only. YouTube OAuth chato em desktop (loopback URI). Beatmaker em Mac fica fora. Cobranca recorrente desktop = licensing manual. Distribuicao via PyInstaller toda atualizacao.

### 2. Web SaaS classico (Next.js + FastAPI + Supabase)
- **Pros:** Stack default Henrique. OAuth callback URL fixa funciona limpo. Multitenancy nativa via RLS. Beatmaker em qualquer SO. Cobranca recorrente (Stripe) trivial quando o momento vier.
- **Contras:** Mais infra pra manter (web + api + db + worker). Custo inicial maior que desktop.

### 3. Hibrido (Next.js + n8n por baixo)
- **Pros:** Reaproveita o n8n que o Gustavo ja construiu. Validacao em 1-2 semanas.
- **Contras:** Debito tecnico imediato. Multitenancy fragil em n8n. Sem pattern claro pra evoluir.

## Decisao

**Veredito: Opcao 2 — Web SaaS classico.**

Custo de manter web > desktop, mas o ganho em alcance (Mac), OAuth e cobranca compensa com folga. Reaproveitamos patterns do `transcribe` (worker async, multi-provider AI, usage_tracker) sem reaproveitar codigo direto — adaptamos pra contexto async/web.

### Stack final

| Camada | Tech | Por que |
|---|---|---|
| Frontend | Next.js 15 + TypeScript + Tailwind + shadcn/ui | Default Henrique, deploy Vercel free tier, App Router maduro |
| Backend | Python 3.11 + FastAPI + Pydantic v2 | Default Henrique, ffmpeg + libs IA bem suportados |
| DB + Auth + Storage | Supabase (Postgres + RLS) | 1 vendor, multitenancy nativo, OAuth Google embutido |
| Worker async | Upstash QStash | HTTP-based, $0 idle, sem precisar Celery+Redis. Workers fazem >10s |
| Audio analysis | Gemini 2.0 Flash | BPM/key/vibe/artistas em 1 chamada. Free tier generoso |
| Tag research | Gemini grounded search | Tags trending sem scrapear, sem banco proprio |
| Geracao copy | Anthropic Claude Sonnet 4.6 | Descricoes ricas, 3 variacoes com angulos forcados distintos |
| Conversao audio | ffmpeg-python + libmp3lame + loudnorm | Padrao da industria |
| Geracao video | ffmpeg (capa.jpg + audio.mp3 → mp4) | Tela estatica, padrao type beat YouTube |
| YouTube | google-api-python-client + OAuth 2.0 | API oficial |
| Deploy web | Vercel | Free tier cobre MVP, integra GitHub |
| Deploy api | Railway | Buildpack Python + ffmpeg nativo |

### Repo: monorepo

Um repo `beatpost` com `web/`, `api/`, `supabase/`. Mais simples pra Gustavo gerenciar. Deploy continua separado (Vercel aponta pra `web/`, Railway aponta pra `api/`).

## Consequencias

- Gustavo aprende Next.js + FastAPI + Supabase juntos. Curva mais ingreme que so n8n, mas alinhado com stack profissional.
- Custo inicial $25-80/mes (Vercel + Railway + Supabase + QStash + Gemini + Claude) — 3 pagantes a $25 ja cobrem.
- Trocar de provider de IA depois e barato (abstracao em `services/`).
- Se um dia precisar desktop, reuso patterns do transcribe.
- Migracao do n8n: nao reaproveitamos. Re-implementamos em Python pra ter testes, observabilidade, RLS.
